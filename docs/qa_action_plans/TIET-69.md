# TIET-69 액션 플랜

- Jira: https://dolomood.atlassian.net/browse/TIET-69
- 주제: 닉네임 검증 실패 시 토스트/키보드 UX 일관성

## QA 결과 요약
- 성공(✅)
  - 회원가입 닉네임 화면의 invalid 제출 시 토스트 노출
  - 완료/확인 제출에서 토스트 동작 안정성
  - 키보드 동작 정책 일관성
- 실패(⚠️)
  - 마이페이지 닉네임 수정 정책이 회원가입 정책과 불일치
  - 12자 초과 경고 미노출
  - 숫자/특수문자 제한 문구 미노출 및 invalid 상태 변경 적용됨

## 원인 가설
- 회원가입(`create`)과 마이페이지(`update`)가 서로 다른 검증 경로를 사용
- `update` 화면은 정책 실패 시 저장 차단/토스트 노출 로직이 submit 단계에서 완전히 통합되지 않았을 가능성
- 공통 검증 함수 부재로 메시지/차단 정책이 분기마다 파편화

## 관련 파일
- `lib/feature/dolo_profile/ui/page/dolo_nickname_create_page.dart`
- `lib/feature/dolo_profile/ui/page/dolo_nickname_update_page.dart`

## 수정 계획
1. 닉네임 정책 공통화
- 길이/문자셋/빈값 검증 함수를 공통 유틸 또는 동일 로직으로 통일합니다.

2. 마이페이지 저장 전 차단 강화
- invalid 결과면 저장 API 호출을 중단하고 토스트를 강제 노출합니다.
- `12자 이내`, `한글/영어만` 메시지를 회원가입과 동일 문구로 맞춥니다.

3. 토스트 위치/표현 정책 통일
- 회원가입/마이페이지 모두 동일 토스트 컴포넌트와 위치 규칙을 사용합니다.

4. 키보드 정책 회귀 확인
- invalid 제출 시 키보드 유지/내림 정책이 기존 합의와 일치하는지 확인합니다.

## 검증 시나리오
- 마이페이지에서 13자 입력 후 제출 시 토스트 노출 + 저장 차단
- 숫자/특수문자 포함 입력 후 제출 시 토스트 노출 + 저장 차단
- 회원가입/마이페이지 각각 동일 입력으로 같은 결과(문구/차단) 확인

## 완료 기준
- TIET-69의 마이페이지 관련 ⚠️ 항목 모두 ✅ 전환
- 회원가입 흐름 기존 성공 항목 회귀 없음
