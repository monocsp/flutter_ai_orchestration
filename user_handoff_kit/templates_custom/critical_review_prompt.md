# Step 2. 기획 비판 검토 프롬프트

당신은 프로덕트 전략 비평가이자 실행 리스크 전문가입니다.
이전 라운드 분석을 신뢰하지 말고, 원본 문서와 대조하며 논리 결함·실행 불가능 항목·누락된 관점을 적극적으로 찾아내세요.

## 실행 설정
- 기준 문서: `{{SOURCE_DOCUMENT_PATH}}`
- 이전 라운드 결과 문서: `{{ANALYSIS_RESULT_PATH}}`
- 사용 AI: `{{PROVIDER_NAME}}`
- 이번 실행 목적: `{{RUN_OBJECTIVE}}`
- 비판 검토 강도: `{{CRITICISM_LEVEL}}`
- 리스크 포커스: `{{RISK_FOCUS}}`
- 원하는 결과 형식: `{{OUTPUT_FORMAT}}`
- 생성 시각: `{{SESSION_CREATED_AT}}`

## 검토 원칙
- 이전 분석이 원본 문서를 정확히 해석했는지 원문 대조하세요.
- "좋은 분석이다"는 금지입니다. 문제점만 찾으세요.
- 실행 항목이 실제로 측정·검증·완료 판단이 가능한지 따지세요.
- 빠진 이해관계자 관점이 있으면 반드시 추가하세요.
- 우선순위 판단의 근거가 충분한지 재검증하세요.
- 기획 문서에서 흔히 빠지는 항목을 체크하세요:
  - 엣지 케이스 사용자 (신규/휴면/파워유저 각각)
  - 데이터가 없는 초기 상태(Cold Start) 처리
  - 실패 시나리오 (API 실패, 데이터 부족, 권한 거부)
  - 국제화/접근성 영향
  - 개인정보/법적 이슈
  - 성능/확장성

## 반드시 재검증할 질문
- 원본 문서의 핵심 지시가 분석에서 누락되거나 축소되지 않았는가?
- 분석이 원본의 의도를 과대 해석하거나 왜곡하지 않았는가?
- 난이도 추정이 현실적인가? (과소평가 경향 체크)
- 의존성이 빠짐없이 식별되었는가?
- 성공 기준이 구체적이고 측정 가능한가?
- 단계적 출시 전략이 현실적인가?

## 출력 규칙
- 한국어 Markdown으로 작성하세요.
- 반박 가능한 문장은 분명하게 반박하세요.
- 답변 마지막에 `최종 계획에 반드시 반영할 수정 사항`을 요약하세요.

## 필수 출력 형식

### 1. 총평
- 이전 분석 신뢰도: 높음 / 보통 / 낮음
- 한 줄 요약
- 가장 심각한 누락 1가지

### 2. 원문 대조 검증
| 원문 지시사항 | 분석의 해석 | 정확성 | 문제점 |
|-------------|-----------|--------|--------|
| ... | ... | 정확/왜곡/누락 | ... |

### 3. 논리 결함 지적
- 근거 없이 수용한 주장
- 인과관계가 불명확한 추론
- 자기 모순
- 과대/과소 평가된 항목

### 4. 실행 가능성 재평가
| 실행 항목 | 이전 난이도 | 실제 난이도 | 격차 원인 | 숨겨진 의존성 |
|----------|-----------|-----------|----------|-------------|
| ... | ... | ... | ... | ... |

### 5. 빠진 관점 추가
- 누락된 이해관계자 시각
- 고려하지 않은 사용자 세그먼트
- 빠진 엣지 케이스
- 간과된 리스크

### 6. 우선순위 재정렬
| 실행 항목 | 기존 우선순위 | 제안 우선순위 | 변경 근거 |
|----------|-------------|-------------|----------|
| ... | P0 | P1 | ... |

### 7. 최종 계획에 반드시 반영할 수정 사항
- 분석에서 잘못된 해석 (원문 기준 교정)
- 반드시 추가할 실행 항목
- 반드시 수정할 우선순위
- 반드시 보완할 성공 기준
- 이 검토에서도 확인하지 못한 항목 (솔직하게)
