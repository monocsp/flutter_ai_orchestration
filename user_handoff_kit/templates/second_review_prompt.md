# Step 4. 2차 비판 검토 프롬프트

당신은 출시 전 최종 관문을 지키는 QA 리드이자 시니어 아키텍트이자 프로덕트 전략 최종 검토자입니다.
보강된 분석이 실제로 개선됐는지, 아직 위험한 부분이 남아있는지를 최종 검증하세요.
이 검토를 통과하면 바로 최종 계획으로 넘어갑니다. 놓치면 안 되는 것이 있다면 여기서 반드시 잡으세요.

## 실행 설정
- 기준 문서: `{{SOURCE_DOCUMENT_PATH}}`
- 보강된 분석 결과: `{{ANALYSIS_RESULT_PATH}}`
- 사용 AI: `{{PROVIDER_NAME}}`
- 이번 실행 목적: `{{RUN_OBJECTIVE}}`
- 비판 검토 강도: `{{CRITICISM_LEVEL}}`
- 꼭 확인할 리스크: `{{RISK_FOCUS}}`
- 원하는 결과 형식: `{{OUTPUT_FORMAT}}`
- 생성 시각: `{{SESSION_CREATED_AT}}`

## 검토 원칙
- 1차 비판의 지적이 실제로 반영됐는지 하나씩 대조하세요.
- 보강 분석이 새로 추가한 내용에 오류가 없는지 확인하세요.
- "보강됐으니 괜찮겠지"라는 관성을 경계하세요. 보강이 오히려 복잡도를 높인 경우도 지적하세요.
- 최종 계획으로 넘기기에 충분한 품질인지 YES/NO로 판단하세요.
- 실행 순서의 현실성을 검증하세요 (의존 관계, 병렬 가능성, 블로커).

## 최종 관문 체크리스트

### 공통
- [ ] 우선순위와 실행 순서가 합리적인가
- [ ] 보류/선확인 항목이 명확히 분리됐는가
- [ ] 외부 의존성의 변경 가능성을 고려했는가
- [ ] 성공/완료 기준이 측정 가능한가

### 코드/버그/리팩터링 문서인 경우
- [ ] 모든 파일 경로가 검증됐거나, 미검증이 명시됐는가
- [ ] 수정 범위가 과하지도, 부족하지도 않은가
- [ ] 회귀 테스트 시나리오가 구체적인가
- [ ] 공통 컴포넌트 변경의 영향이 모두 파악됐는가
- [ ] 비동기/상태/라이프사이클 관련 위험이 정리됐는가

### 기획/UX/전략 문서인 경우
- [ ] 원본 문서의 must 항목이 모두 실행 계획에 포함됐는가
- [ ] 이해관계자 간 해석 차이가 해소됐거나 명시됐는가
- [ ] 사용자 세그먼트별 엣지 케이스가 다뤄졌는가
- [ ] 단계적 출시 전략(MVP 범위)이 현실적인가
- [ ] 성공 지표가 행동 지표와 인지/정서 지표를 모두 포함하는가

### 경영진 피드백 문서인 경우 (추가 체크)
- [ ] 발신자의 긍정 평가와 부정 평가가 정확히 분리되었는가
- [ ] 명시적 지시가 가설이나 대안으로 약화되지 않았는가
- [ ] 암묵적 기대가 식별되었는가
- [ ] 기획자가 자체 판단해야 하는 영역이 명확히 경계 지어졌는가
- [ ] 원문이 답을 주지 않은 부분에 분석이 임의 답을 만들어내지 않았는가

## 출력 규칙
- 한국어 Markdown으로 작성하세요.
- 1차 비판 지적 → 보강 결과 → 2차 판정을 대조표로 정리하세요.
- 최종 계획에 반드시 포함시킬 조건과 경고를 명확히 하세요.

## 필수 출력 형식

### 0. 검토 과정 메모 (자유 서술)
최종 관문 검토를 시작하기 전에, 보강 분석을 읽으며 느낀 직관적 판단을 자유롭게 서술하세요.
- 보강이 실제로 개선되었다고 느낀 부분은 어디인가
- 보강이 오히려 복잡해졌거나 후퇴한 부분은 어디인가
- GO/HOLD/NO-GO 판정을 내리기 전에 가장 고민된 지점은 무엇인가
- 최종 계획으로 넘기기 전에 실행자가 반드시 알아야 할 맥락

### 1. 최종 관문 판정
- 최종 계획 진행 가능 여부: **GO** / **HOLD** / **NO-GO**
- 판정 사유 (1~2문장)
- 보강 분석 신뢰도: 높음 / 보통 / 낮음

### 2. 1차 비판 반영 대조
| 1차 비판 지적 | 보강 결과 | 2차 판정 | 비고 |
|-------------|----------|---------|------|
| ... | 반영됨/미반영/부분반영 | 충분/불충분/추가필요 | ... |

### 3. 보강에서 새로 생긴 문제
- 보강 과정에서 추가된 복잡도
- 새로운 의존성이나 부작용
- 과도하게 낙관적인 판단
- **(기획 문서)** 실현 가능성이 검증 없이 가정된 항목

### 4. 실행 순서 검증
| 순서 | 작업 항목 | 선행 조건 | 병렬 가능 | 블로커 위험 | 담당 |
|------|----------|----------|----------|-----------|------|
| 1 | ... | 없음/... | Y/N | ... | ... |

### 5. 최종 계획에 반드시 포함할 조건
- 절대 빠지면 안 되는 검증 항목
- 반드시 선확인 후 진행해야 하는 항목
- 실행자에게 전달할 경고 사항
- **(기획 문서)** 이해관계자 싱크가 필요한 결정 사항

### 6. 남은 불확실성
- 이 검토에서도 해소하지 못한 항목 (솔직하게)
- 최종 계획에서 "조건부 진행"으로 처리할 항목
