# Step 5. 최종 종합 계획 프롬프트

당신은 최종 의사결정을 내리는 기술 리드이자 프로덕트 오너입니다.
기준 문서와 이전 라운드의 모든 분석/비판 검토를 종합해, 실행자가 바로 착수할 수 있는 최종 실행 계획서를 작성하세요.
추상적 구호가 아닌, "이 문서만 보고 내일 아침부터 작업 시작"이 가능한 수준이어야 합니다.

## 중요: 출력 규칙
- 최종 계획서의 전체 내용을 이 응답에 직접 출력하세요.
- 파일에 저장하거나 별도 경로에 쓰지 마세요. 반드시 이 응답 본문에 전문을 포함하세요.
- 요약만 출력하는 것은 금지입니다. 모든 섹션을 빠짐없이 작성하세요.

## 실행 설정
- 기준 문서: `{{SOURCE_DOCUMENT_PATH}}`
- 이전 라운드 분석 결과: `{{ANALYSIS_RESULT_PATH}}`
- 이전 라운드 비판 검토 결과: `{{CRITICAL_REVIEW_RESULT_PATH}}`
- 사용 AI: `{{PROVIDER_NAME}}`
- 이번 실행 목적: `{{RUN_OBJECTIVE}}`
- 비판 검토 강도: `{{CRITICISM_LEVEL}}`
- 꼭 확인할 리스크: `{{RISK_FOCUS}}`
- 원하는 결과 형식: `{{OUTPUT_FORMAT}}`
- 생성 시각: `{{SESSION_CREATED_AT}}`

## 종합 원칙
- 이전 라운드 간 충돌한 주장들은 `채택 / 기각 / 보류`로 정리하세요.
- 근거가 더 강한 쪽을 우선 채택하세요.
- 원본 문서의 must 항목은 반드시 실행 계획에 반영하세요.
- 모호한 지시는 "이렇게 해석했다"를 명시하고, 실행자가 확인할 수 있게 하세요.
- 의존 관계를 고려한 바로 착수 가능한 순서를 제시하세요.
- 검증 계획 없는 낙관적 결론은 금지합니다.
- 보류 항목은 왜 보류인지, 누가 언제 결정하는지 명시하세요.

## 문서 유형별 필수 포함 관점

### A. 코드/버그/리팩터링 문서인 경우
- 직접 수정 파일, 간접 영향 파일
- 공통 컴포넌트, 상태 관리, 라우팅/라이프사이클 영향
- 번역/문구/디자인 토큰 영향
- 자동 테스트 및 수동 검증 포인트
- 외부 의존성 (패키지, SDK, 서드파티 서비스)

### B. 기획/UX/전략 문서인 경우
- 사용자 경험 흐름 (화면 단위 또는 시나리오 단위)
- 필요한 데이터/인프라 요구사항
- 디자인 산출물 목록
- 개발 구현 범위 추정
- 단계적 출시 전략 (MVP → V1 → V2)
- 성공 지표 (행동 지표 + 인지/정서 지표)

### B-1. 경영진 피드백 문서인 경우 (B에 추가로 포함)
- **원문 긍정 평가 정리**: 유지해야 할 방향과 인정받은 부분을 명확히
- **원문 부정 평가/수정 지시 정리**: 반드시 바꿔야 할 부분과 수정 강도
- **발신자의 의도 해석**: 이 피드백이 궁극적으로 원하는 변화
- **기획자 행동 가이드**: 자체 판단 영역 vs 상위 확인 필요 영역 분리
- **핵심 요구사항 대조표**: 원문 지시 → 실행 계획 항목 매핑 (누락 없는지 검증)

### 공통
- 보류 또는 선확인 항목
- 이해관계자 확인 필요 항목 (기획, 디자인, 정책, 인프라)
- QA/검증 시나리오

## 출력 규칙
- 한국어 Markdown으로 작성하세요.
- 바로 실행 가능한 순서로 정리하세요.
- 각 항목에 담당 역할을 명시하세요.
- 추상적 구호보다 구체적 행동, 파일, 산출물, 검증 순서를 적으세요.
- 답변 마지막에 `실행자가 첫 30분(또는 첫 3일) 안에 볼 것`을 짧게 정리하세요.

## 필수 출력 형식

### 0. 종합 과정 메모 (자유 서술)
최종 계획을 작성하기 전에, 이전 라운드를 종합하며 내린 핵심 판단의 맥락을 자유롭게 서술하세요.
- 이전 라운드 간 가장 크게 충돌한 주장과 어느 쪽을 택했는지, 왜
- 최종 계획에서 가장 자신 있는 판단과 가장 불안한 판단
- 실행자가 이 계획을 읽기 전에 알아야 할 배경 맥락
- 이 계획서의 한계: 여기서 다루지 못한 것, 추후 보완이 필요한 것

### 1. 최종 판단
- 작업 성격: 버그 수정 / 기능 추가 / 정책 변경 / 리팩터링 / UX 개선 / 방향 수정 / 복합
- 최종 판단 요약 (3문장 이내)
- 총 예상 규모: 소(1~2일) / 중(3~5일 또는 1~2주) / 대(1주 또는 1개월 이상)

### 2. 의사결정 정리
| 쟁점 | 채택안 | 기각안 또는 보류안 | 판단 근거 |
|------|--------|------------------|----------|
| ... | ... | ... | ... |

### 3. 실행 로드맵
| 단계 | 기간 | 핵심 산출물 | 완료 조건 | 의존성 |
|------|------|-----------|----------|--------|
| Phase 1 (MVP/즉시수정) | ... | ... | ... | 없음 |
| Phase 2 (단계적적용) | ... | ... | ... | Phase 1 완료 |
| Phase 3 (구조개선) | ... | ... | ... | Phase 2 완료 |

### 4. 최종 작업 보드
| 순서 | 작업 항목 | 담당 | 상태 | 난이도 | 핵심 파일/산출물 | 목적 | 선행 조건 |
|------|----------|------|------|--------|---------------|------|----------|
| 1 | ... | 개발/기획/디자인/QA | 즉시착수 | 소/중/대 | ... | ... | 없음 |

상태 분류:
- **즉시착수**: 바로 착수 (즉시수정/즉시기획)
- **확인후착수**: 선행 확인 완료 후
- **단계적적용**: 선행 작업 완료 후
- **구조개선**: 기능 완료 후 리팩터링/개선
- **선확인**: 기획/디자인/정책 확인 후 판단
- **보류**: 현재 스코프 밖, 별도 티켓

### 5. 작업 항목별 최종 실행 계획
각 항목마다 아래를 포함하세요.
- **목표**: 이 작업이 완료되면 뭐가 달라지는지
- **구체적 행동**: 누가 무엇을 어떻게 하는지
- **(코드 문서)** 직접 수정 파일, 간접 영향 파일, 구체적 변경 방향
- **(기획 문서)** 필요 산출물 (와이어프레임, 기획서, API 스펙, 디자인 시안 등)
- **주의할 리스크/회귀 포인트**
- **완료 조건**: 측정 가능한 조건
- **모호한 부분**: 실행자가 추가 확인해야 할 것

### 6. 성공 지표 (기획/전략 문서인 경우)
| 지표 유형 | 지표명 | 현재 추정값 | 목표값 | 측정 방법 |
|----------|--------|-----------|--------|----------|
| 행동 지표 | ... | ... | ... | ... |
| 인지/정서 지표 | ... | ... | ... | 설문/인터뷰 |

### 7. 검증 계획
- **(코드 문서)** 자동 테스트: 추가하거나 수정할 테스트 목록
- **(코드 문서)** 수동 검증: 화면별, 시나리오별 확인 항목
- **(기획 문서)** 프로토타입 검증: 무엇을 어떤 방법으로 테스트하는지
- **(기획 문서)** 사용자 테스트 시나리오: 구체적 태스크 목록
- 회귀/부작용 확인 순서

### 8. 보류 또는 선확인 항목
| 항목 | 사유 | 담당 | 확인 기한 | 미확인 시 영향 |
|------|------|------|----------|--------------|
| ... | 기획 미확정 | PM/기획자 | ... | 개발 착수 지연 |

### 9. 실행자가 가장 먼저 할 것
- **(코드 문서)** 먼저 열어볼 파일 (3개 이내), 먼저 검증할 가정
- **(기획 문서)** 먼저 확정해야 할 결정 (3개 이내), 먼저 만들어야 할 산출물
- 함정이 되는 포인트
- 모르면 물어봐야 할 사람/채널
