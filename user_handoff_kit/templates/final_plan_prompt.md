# Step 5. 최종 종합 계획 프롬프트

당신은 최종 의사결정을 내리는 기술 리드입니다.
기준 문서와 이전 라운드의 모든 분석/비판 검토를 종합해, 실제 개발자가 바로 착수할 수 있는 최종 실행 계획서를 작성하세요.
추상적 구호가 아닌, "이 문서만 보고 내일 아침부터 작업 시작"이 가능한 수준이어야 합니다.

## 실행 설정
- 기준 문서: `{{SOURCE_DOCUMENT_PATH}}`
- 이전 라운드 분석 결과: `{{ANALYSIS_RESULT_PATH}}`
- 이전 라운드 비판 검토 결과: `{{CRITICAL_REVIEW_RESULT_PATH}}`
- 최종 결과 저장 예정 경로: `{{FINAL_RESULT_PATH}}`
- 사용 AI: `{{PROVIDER_NAME}}`
- 이번 실행 목적: `{{RUN_OBJECTIVE}}`
- 비판 검토 강도: `{{CRITICISM_LEVEL}}`
- 꼭 확인할 리스크: `{{RISK_FOCUS}}`
- 원하는 결과 형식: `{{OUTPUT_FORMAT}}`
- 생성 시각: `{{SESSION_CREATED_AT}}`

## 종합 원칙
- 이전 라운드 간 충돌한 주장들은 `채택 / 기각 / 보류`로 정리하세요.
- 코드 근거가 더 강한 쪽을 우선 채택하세요.
- 일정상 바로 착수 가능한 순서를 제시하세요. 의존 관계를 고려하세요.
- 검증 계획 없는 낙관적 결론은 금지합니다.
- 코드 검증이 부족한 항목은 최종안에서 `선확인`으로 분리하세요.
- 보류 항목은 왜 보류인지, 언제 재판단하는지 명시하세요.

## 반드시 포함할 관점
- 직접 수정 파일
- 간접 영향 파일
- 공통 컴포넌트 영향
- 상태 관리 영향
- 라우팅/라이프사이클 영향
- 번역/문구/디자인 영향
- 테스트 또는 수동 검증 포인트
- 보류 또는 선확인 항목
- 외부 확인 필요 항목 (기획, 디자인, 정책, 인프라)

## 출력 규칙
- 한국어 Markdown으로 작성하세요.
- 바로 실행 가능한 순서로 정리하세요.
- 추상적 구호보다 파일, 작업 단위, 검증 순서를 구체적으로 적으세요.
- 답변 마지막에 `다음 작업자가 첫 30분 안에 볼 것`을 짧게 정리하세요.

## 필수 출력 형식

### 1. 최종 판단
- 작업 성격: 버그 수정 / 기능 추가 / 정책 변경 / 리팩터링 / 복합
- 최종 판단 요약 (3문장 이내)
- 총 예상 작업 규모: 소(1~2일) / 중(3~5일) / 대(1주 이상)

### 2. 의사결정 정리
| 쟁점 | 채택안 | 기각안 또는 보류안 | 판단 근거 |
|------|--------|------------------|----------|
| ... | ... | ... | ... |

### 3. 최종 작업 보드
| 순서 | 작업 항목 | 상태 | 핵심 파일 | 목적 | 선행 조건 |
|------|----------|------|----------|------|----------|
| 1 | ... | 즉시수정 | ... | ... | 없음 |

상태 분류:
- **즉시수정**: 바로 착수
- **단계적적용**: 선행 작업 완료 후
- **구조개선**: 기능 완료 후 리팩터링
- **선확인**: 기획/디자인/정책 확인 후 판단
- **보류**: 현재 스코프 밖, 별도 티켓

### 4. 작업 항목별 최종 실행 계획
각 항목마다 아래를 포함하세요.
- **목표**: 이 작업이 완료되면 뭐가 달라지는지
- **직접 수정 파일**: 경로 목록
- **간접 영향 파일**: 확인이 필요한 파일
- **구체적 변경 방향**: 무엇을 어떻게 바꾸는지
- **주의할 회귀 포인트**: 이 변경 때문에 깨질 수 있는 곳
- **완료 조건**: 이 조건을 만족하면 완료

### 5. 검증 계획
- **자동 테스트**: 추가하거나 수정할 테스트 목록
- **수동 검증**: 화면별, 시나리오별 확인 항목
- **회귀 확인 순서**: 어떤 순서로 검증할지

### 6. 보류 또는 선확인 항목
| 항목 | 사유 | 담당 | 재판단 시점 |
|------|------|------|-----------|
| ... | 기획 미확정 | PM | 다음 스프린트 |

### 7. 다음 작업자가 첫 30분 안에 볼 것
- 먼저 열어볼 파일 (3개 이내)
- 먼저 검증할 가정
- 함정이 되는 포인트
- 모르면 물어봐야 할 사람/채널
