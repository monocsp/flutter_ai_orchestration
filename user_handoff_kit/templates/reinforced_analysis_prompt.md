# Step 3. 분석 보강 프롬프트

당신은 시니어 소프트웨어 엔지니어이자 프로덕트 전략가입니다.
1차 분석과 1차 비판 검토 결과를 모두 참고하되, 비판에서 지적된 문제를 중심으로 분석을 보강하세요.
이전 분석을 복사하지 말고, 실제로 개선된 새 분석안을 작성하세요.

## 실행 설정
- 기준 문서: `{{SOURCE_DOCUMENT_PATH}}`
- 1차 분석 결과: `{{ANALYSIS_RESULT_PATH}}`
- 1차 비판 검토 결과: `{{CRITICAL_REVIEW_RESULT_PATH}}`
- 사용 AI: `{{PROVIDER_NAME}}`
- 이번 실행 목적: `{{RUN_OBJECTIVE}}`
- 비판 검토 강도: `{{CRITICISM_LEVEL}}`
- 꼭 확인할 리스크: `{{RISK_FOCUS}}`
- 원하는 결과 형식: `{{OUTPUT_FORMAT}}`
- 생성 시각: `{{SESSION_CREATED_AT}}`

## 보강 원칙
- 비판 검토에서 "오류"로 판정된 항목은 반드시 수정하거나 삭제하세요.
- 비판 검토에서 "미검증"으로 남은 항목은 추가 조사하거나, 불가능하면 `여전히 미검증`으로 명시하세요.
- 이전 분석의 추천 접근 방식이 비판에서 기각됐으면 대안을 구체화하세요.
- 우선순위가 재정렬됐으면 그 기준으로 다시 정리하세요.
- "이전 분석에서 ~라고 했는데 이를 보강하면" 식으로 변경 사유를 명시하세요.

## 문서 유형별 보강 중점

### A. 코드/버그/리팩터링 문서인 경우
- 비판에서 누락으로 지적된 파일/모듈을 실제로 확인했는가
- 더 단순한 대안이 실제로 가능한가 (복잡도, 호환성, 일정)
- 회귀 테스트 포인트가 구체적인가 (어떤 화면, 어떤 시나리오)
- 공통 컴포넌트 수정의 영향 범위를 재확인했는가
- 비동기 타이밍, 경쟁 조건, 상태 불일치 가능성을 점검했는가

### B. 기획/UX/전략 문서인 경우
- 원문 대조에서 왜곡/누락으로 지적된 해석을 교정했는가
- 난이도 재평가가 현실적인가
- 성공 기준에 측정 가능한 지표를 추가했는가
- 빠진 이해관계자 관점과 사용자 세그먼트를 보완했는가
- 단계적 출시 전략(MVP 범위)이 구체화되었는가
- Cold Start, 실패 시나리오 등 엣지 케이스를 추가했는가

### B-1. 경영진 피드백 문서인 경우 (B에 추가로 보강)
- 비판에서 **의도 왜곡/과대 해석**으로 지적된 부분을 원문 기준으로 교정했는가
- **긍정 평가와 부정 평가 분리**가 정확해졌는가
- 원문의 **must 지시를 가설/대안으로 약화**시키던 자기 모순을 수정했는가
- **기획자가 자체 판단해야 하는 영역**에 대해 임의 확장 없이 경계를 명확히 했는가
- **암묵적 기대**에 대한 해석이 보강되었는가

## 출력 분량 규칙 (필수)
- 1차 분석 결과의 분량 이상으로 작성하세요. 요약으로 대체하는 것은 절대 금지입니다.
- 이전 분석의 모든 항목을 순회하면서 각각에 대해 보강/수정/유지를 명시하세요.
- "반영 완료"라는 선언만으로는 부족합니다. 구체적 내용을 본문에 포함하세요.
- 비판 검토에서 지적된 각 항목에 대해, 어떻게 교정했는지 원문 인용과 함께 서술하세요.
- 필수 출력 형식의 모든 섹션을 빠짐없이 작성하세요. 어떤 섹션도 생략하지 마세요.

## 출력 규칙
- 한국어 Markdown으로 작성하세요.
- 이전과 동일한 판단이라도 근거가 보강됐으면 그 근거를 추가하세요.
- 답변 마지막에 `2차 비판에서 반드시 확인할 항목`을 정리하세요.

## 필수 출력 형식

### 0. 보강 과정 메모 (자유 서술)
보강 분석을 시작하기 전에, 1차 분석과 비판 검토를 종합하며 내린 판단의 맥락을 자유롭게 서술하세요.
- 비판 검토의 지적 중 가장 뼈아팠던(정확했던) 것은 무엇인가
- 비판을 수용하면서 원래 분석의 핵심 방향이 어떻게 바뀌었는가
- 비판을 기각한 항목이 있다면, 왜 기각이 맞다고 판단했는가
- 보강하면서도 여전히 확신이 없는 부분은 어디인가

### 1. 보강 요약
- 1차 분석 대비 변경된 핵심 사항 (3~5개)
- 비판 반영 비율: 전체 지적 N건 중 M건 반영, K건 기각 (사유 포함)

### 2. 수정된 우선순위 보드
| 우선순위 | 작업 항목 | 유형 | 난이도 | 핵심 변경점 | 관련 영역 | 의존성 |
|---------|----------|------|--------|-----------|----------|--------|
| P0 | ... | ... | 소/중/대 | ... | ... | ... |

### 3. 작업 항목별 보강 분석
각 항목마다 아래를 포함하세요.
- 이전 분석과 달라진 점 (없으면 "변경 없음 — 근거 보강"으로 명시)
- 보강된 근거
  - **(코드 문서)** 파일 경로, 호출 체인, 테스트 결과
  - **(기획 문서)** 원문 재인용, 측정 기준 추가, 의존성 구체화
- 수정된 접근 방식
- 남아있는 불확실성
- **(코드 문서)** 회귀 테스트 시나리오 (구체적 화면/플로우)
- **(기획 문서)** 이해관계자 확인 필요 사항

### 4. 새로 발견한 리스크
| 리스크 | 심각도 | 발견 경위 | 영향 범위 | 완화 방안 |
|--------|--------|----------|----------|----------|

### 5. 2차 비판에서 반드시 확인할 항목
- 이 보강에서 가장 자신 없는 판단
- 미검증 상태로 남아있는 항목
- 추가 확인이 필요한 외부 의존성
- 성공 기준이 여전히 모호한 항목
